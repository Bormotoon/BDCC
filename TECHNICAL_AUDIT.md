## 3. Архитектура

### 3.1 Точка входа и управление

**MainScene.gd** (2375 строк) — центральный объект всей игры. Управляет:
- Стеком сцен
- Временем (дни, недели, время суток)
- Флагами (глобальные, модульные, датапаковые)
- Пулом динамических NPC
- Сохранением/загрузкой
- Загрузкой модов и датапаков
- Отладочными действиями (цепочка if/elif ~200 строк)
- Оркестрацией UI

**GlobalRegistry.gd** (3018 строк) — второй God Object. Центральный реестр для ~50 типов сущностей (сцены, персонажи, части тела, предметы, навыки, перки, секс-активности, трансформации, датапаки). Каждый тип следует шаблону `registerX()` / `registerXFolder()` / `createX()` / `getXRef()`.

### 3.2 Паттерны зависимости

Вся архитектура построена на **глобальных синглтонах** `GM.*`:
- `GM.main` — MainScene
- `GM.pc` — игрок (Player)
- `GM.world` — мир (World)
- `GM.ui` — интерфейс
- `GM.ES` — EventSystem
- `GM.IS` — InteractionSystem
- `GM.RS` — RelationshipSystem

Это создаёт **сильную связанность** (tight coupling) между всеми подсистемами.

### 3.3 Иерархия персонажей

```
BaseCharacter (3918 строк) — базовый класс для всех
  └── Player (1019 строк) — расширяет игрока
  └── Character (Characters/) — NPC
      └── DynamicCharacter — случайно генерируемые NPC
```

**BaseCharacter** использует **композицию**: держит Inventory, BuffsHolder, SkillsHolder, LustInterests, FetishHolder, Personality, Fluids, Orifices. Это хорошее решение, но секс-механика (~400 строк) встроена внутрь и следовало бы вынести.

### 3.4 Мир и навигация

**World.gd** (757 строк) — 2D сеточный мир с комнатами, A* pathfinding (Godot AStar2D), затемнением, зонами. Чистая архитектура. Pawns (NPC-сущности) рендерятся поверх карты.

### 3.5 Interaction System

**InteractionSystem.gd** (728 строк) — система имитации жизни NPC. Управляет pawns, взаимодействиями, глобальными задачами. Использует time-slicing (макс. 10 тиков по 60 секунд) для производительности. Хорошая развязка от MainScene.

### 3.6 Sex Engine

**SexEngine.gd** (1854 строки) — сложнейшая подсистема. Построена как конечный автомат (state machine) с дом/sub ролями, генерацией целей, динамическими joiner'ами, оценкой активностей, AI-принятием решений, приоритетами анимаций. Структура `SexDomInfo`/`SexSubInfo` чистая. Включает поддержку тройничков (threesomes) с динамическим присоединением.

### 3.7 Модульная система

**Module.gd** (191 строк) — базовый класс для модулей. 22 модуля контента:

| Модуль | Назначение |
|---|---|
| TaviModule, RahiModule, JackiModule, SocketModule, ArticaModule, AlexRynardModule, KaitModule, NovaModule | Маршруты NPC (сюжет) |
| FightClubModule, GymModule, MedicalModule, CellblockModule | Локации |
| NpcSlaveryModule, PlayerSlaveryModule, SlaveAuctionModule | Рабство |
| HypnokinkModule, PortalPantiesModule, PunishmentsModule | Спец. системы |
| DrugDenModule | Рогалик-данжон |
| ElizaModule, AcePregExpac, SongJoHairsModule | Контент |

Каждый модуль сам регистрирует свои сущности в `GlobalRegistry`.

## 4. Ключевые подсистемы

### 4.1 Datapack System (Datapack.gd, 348 строк)
Система пользовательского контента. Позволяет создавать персонажей, скины, сцены, квесты **без GDScript**. Включает визуальный редактор сцен (CrotchCode — 196 блоков, 11,143 строк) — это полноценный визуальный язык программирования для моддинга.

### 4.2 CrotchCode — визуальный язык
196 типов блоков: математика, логика, флаги, переменные, NPC, инвентарь, lewd-действия, квесты, анимации, строки, изображения, RNG. `CodeContex.gd` (727 строк) — исполнительный движок.

### 4.3 3D Rendering (Doll3D.gd, 963 строки)
Модульная 3D-кукла: части тела слотятся динамически. Масштабирование костей для беременности, груди, ягодиц. Jiggle-физика. Частицы (семя, молоко). Цепи/наручники между куклами. Тела-надписи (body writings).

### 4.4 Sex Toy Integration (Util/SexToySupport/)
3 бэкенда: **Buttplug.IO** (WebSocket v4, 366 строк), **Lovense Connect** (414 строк), **XToys.App** (157 строк). Игровые события маппятся на паттерны вибрации через кривые интенсивности.

### 4.5 Автоперевод (Util/AutoTranslation/)
Цепочка переводчиков с фоллбэком: Google, GoogleBatch, Papago (Naver), Microsoft/Bing. Разбивка текста на 4000-символьные чанки с задержкой.

### 4.6 In-Game Computer (Game/Computer/)
Эмулятор командной строки с сетевой симуляцией: серверы с IP, логин/аутентификация, файловые системы, `ls`/`cat`/`connect`/`login`. Включает easter egg `cat tunya`.

### 4.7 Системы шейдеров (Shaders/)
5 простых canvas_item шейдеров (~150 строк суммарно): тень, контур, умный контур, аура, силуэт. 3D использует встроенный рендеринг Godot.

### 4.8 Pregnancy & Transformation
Полная система беременности с менструальным циклом, яйцеклетками, совместимостью видов. Система трансформаций с 20+ зельями, градуальными изменениями, composable TF, полной обратимостью.

## 6. Критические проблемы

### 6.1 God Objects (нарушение SRP)
- **MainScene.gd** (2375 строк): делает слишком много. Debug action — монстрозная цепочка if/elif. Флаги маршрутизируются через строковый парсинг ("ModuleID.FlagID").
- **GlobalRegistry.gd** (3018 строк): колоссальное дублирование кода (register/create/registerFolder для каждого типа). Типобезопасность отсутствует — всё через строки.
- **BaseCharacter.gd** (3918 строк): секс-механика (~400 строк) встроена в базовый класс персонажа.

### 6.2 Глобальное состояние
`GM.*` синглтоны создают неявные зависимости между всеми подсистемами. Любой скрипт может обратиться к `GM.main`, `GM.pc`, `GM.world` и т.д. Это делает тестирование невозможным и создаёт хрупкость.

### 6.3 Типобезопасность
389 зарегистрированных классов, но взаимодействие между ними часто через строки: ID предметов, ID сцен, ID флагов, имена костей. Одна опечатка = silent bug.

### 6.4 Отсутствие тестов
**Нет ни одного автоматизированного теста.** Для проекта с 3000+ файлами GDScript и 389 классами это критический риск. Единственный "тест" — `SimpleStringInterpolatorTester.gd` — это ручной UI-тестер, а не автоматизированный тест.

### 6.5 Размер спрайта (Godot 3.x limitation)
`_global_script_classes` — массив из 389 элементов, вручную синхронизируемых. При добавлении/удалении класса нужно вручную править project.godot. В Godot 4.x это автоматизировано.

### 6.6 Устаревший движок
Godot 3.x:
- Нет GDExtension (C++/Rust расширения)
- Нет Vulkan-рендера
- GDScript без статической типизации
- HTML5 экспорт deprecated
- Ограниченная поддержка Android/iOS
- Нет backward compatibility с Godot 4.x

### 6.7 Нет статического анализа
Нет линтеров, type checker'ов, CI/CD. Качество кода обеспечивается только ревьюми и ручным тестированием.

## 7. Фундаментальные проблемы

### 7.1 Технологический долг
Проект привязан к Godot 3.x. Миграция на Godot 4.x потребует:
- Замены `Reference` → `RefCounted`
- Обновления GDScript синтаксиса
- Переработки `_global_script_classes`
- Обновления шейдеров (GLSL → Godot Shading Language 2.0)
- Проверки всех 70 3D-сцен
- Проверки всех 196 CrotchCode-блоков

Это **месяцы работы** для активного проекта.

### 7.2 Масштабируемость архитектуры
- 1005 файлов в `Game/` — нет чёткой модульной границы
- MainScene — единственный точка входа для всей логики
- GlobalRegistry — единственный реестр для всех типов
- Флаги — строковые ключи без валидации

### 7.3 Производительность
- 3000+ GDScript файлов = долгое время загрузки
- `GlobalRegistry` загружает всё в память при старте
- 3D-сцены для каждого типа взаимодействия (70 штук)
- Нет профилирования (MyProfiler существует, но не используется системно)

### 7.4 Безопасность модов
Датапаки загружаются без sandboxing. CrotchCode выполняет произвольную логику. Нет валидации входных данных от пользователей.

## 8. Положительные стороны

1. **Впечатляющий объём контента**: 3000+ файлов, 22 модуля, 80k+ слов текста в одном обновлении
2. **Хорошая модульность модулей**: каждый модуль автономен, саморегистрируется
3. **CrotchCode** — гениальное решение для моддинга без кода (196 блоков, визуальный редактор)
4. **Interaction System** — хорошо спроектированная NPC-симуляция с time-slicing
5. **3D-кукла** — модульная система с динамическими частями тела
6. **Полная кросс-платформенность** (5 платформ)
7. **Открытый исходный код** с активным комьюнити (PR от множества контрибьюторов)
8. **Datapack browser** — встроенный магазин модов

## 10. Заключение

BDCC — это **впечатляюще масштабный** solo/indie-проект: 3000+ файлов GDScript, 389 классов, 22 модуля контента, 5 целевых платформ, визуальный язык программирования для моддинга, интеграция с физическими секс-игрушками. Проект активно развивается 4+ года.

Однако **технологический фундамент устарел** (Godot 3.x), архитектура страдает от двух God Objects и глобального состояния, а отсутствие тестов создаёт накопительный риск. Миграция на Godot 4.x — необходимость, но это огромная работа для проекта такого масштаба.
